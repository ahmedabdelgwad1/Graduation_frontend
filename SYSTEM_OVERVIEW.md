# 🏛️ نظرة نظرية على النظام — متحف مكتبة الإسكندرية
### موجَّه لمتخصص Data Science

---

## ما الذي بنيناه؟

نظام ذكاء اصطناعي مدمج داخل موقع متحف رقمي، يتيح للزائر أن **يتحدث أو يكتب** سؤالًا عن أي قطعة أثرية ويحصل على إجابة ذكية تعتمد على قاعدة بيانات المتحف الحقيقية — لا على معرفة عامة للنموذج.

---

## 1️⃣ المكونات الكبرى للنظام

```
الزائر / المشرف
       │
       ▼
  [ واجهة المستخدم (Frontend) ]
  موقع Next.js — يعرض القطع والصفحات
  ويحتوي على نافذة دردشة مع الذكاء الاصطناعي
       │
       ▼
  [ خادم الذكاء الاصطناعي (FastAPI Backend) ]
  يستقبل الأسئلة، يبحث في قاعدة المعرفة،
  ويُنتج الإجابات
       │
     ┌─┴───────────────────┐
     ▼                     ▼
[ Supabase ]         [ Vector Store ]
قاعدة بيانات         قاعدة التشابه الدلالي
بيانات القطع         (Chroma الآن → pgvector لاحقًا)
```

## 2️⃣ كيف يعمل الذكاء الاصطناعي؟ — نمط RAG

النظام يعتمد على تقنية اسمها **RAG — Retrieval Augmented Generation**، وهي نمط معياري في أنظمة الـ LLM الحديثة.

### المشكلة التي يحلها RAG

النماذج اللغوية الكبيرة (مثل GPT أو Gemini أو LLaMA) تعرف الكثير عن العالم — لكنها **لا تعرف شيئًا** عن محتوى متحفنا تحديدًا.  
الحل ليس إعادة تدريب النموذج (تكلفة ضخمة جدًا)، بل **نحقن البيانات في وقت الاستدلال (Inference Time).**

### خطوات RAG بالتفصيل

```
السؤال: "ما هو عمر قناع توت عنخ آمون؟"
         │
         ▼
 ① تحويل السؤال إلى Embedding
    النص → نقطة في فضاء متعدد الأبعاد (vector)
    مثلًا: [0.23, -0.87, 0.14, ...]  بحجم 1536 بُعد
         │
         ▼
 ② البحث في Vector Store عن أقرب القطع للسؤال
    (Semantic Search بدل Keyword Search)
    النتيجة: أقرب 5 قطع أثرية لها علاقة بالسؤال
         │
         ▼
 ③ سحب البيانات التفصيلية لهذه القطع من Supabase
    الاسم، الوصف، الحقبة الزمنية...
         │
         ▼
 ④ بناء Prompt غني للـ LLM يحتوي على:
    - سياق المتحف (System Prompt)
    - معلومات القطع المسحوبة (Context)
    - سؤال الزائر
         │
         ▼
 ⑤ LLM يولّد إجابة مبنية على البيانات الحقيقية
    وليس على تخيلاته العامة (Hallucination مقلل)
```

## 3️⃣ Embeddings — قلب النظام

### ما هو الـ Embedding؟

تحويل النص (أو الصوت أو الصورة) إلى **متجه رقمي** في فضاء متعدد الأبعاد بحيث:

- النصوص المتشابهة في المعنى → نقاط قريبة من بعض
- النصوص المختلفة في المعنى → نقاط بعيدة

```
"قناع توت عنخ آمون"  → [0.23, -0.87, 0.14, 0.91, ...]
"mask of tutankhamun" → [0.24, -0.86, 0.13, 0.90, ...]  ← قريب جدًا!
"وصفة طبخ الكشري"   → [0.91,  0.42, 0.78, 0.01, ...]  ← بعيد جدًا
```

### كيف ننتج الـ Embeddings في مشروعنا؟

نستخدم نموذج embedding جاهز (مثل `text-embedding-3-small` من OpenAI، أو نموذج Groq).  
نُدخل وصف كل قطعة أثرية → نحصل على متجه → نحفظه في Vector Store.

هذه العملية تحدث **مرة واحدة** (أو عند إضافة قطع جديدة) عبر سكريبت اسمه `index_artifacts.py`.

## 4️⃣ Vector Store — مقارنة Chroma vs pgvector

### ما هو الـ Vector Store؟

قاعدة بيانات متخصصة في تخزين المتجهات والبحث فيها بكفاءة. البحث فيها ليس بالـ `WHERE` التقليدي، بل بمقياس التشابه الهندسي.

### مقاييس التشابه الشائعة

| المقياس | المعنى | متى يُستخدم |
|--------|--------|------------|
| **Cosine Similarity** | زاوية بين المتجهين | أكثر استخدامًا في NLP |
| **Euclidean Distance** | المسافة المباشرة | حين الحجم مهم |
| **Dot Product** | الضرب النقطي | حين المتجهات مُوحَّدة |

نحن نستخدم **Cosine Similarity** — لأن ما يهمنا اتجاه المعنى وليس حجم النص.

---

### Chroma (الوضع الحالي)

```
المميزات:
✅ سهل الإعداد جدًا (pip install chromadb)
✅ يعمل محليًا بدون internet
✅ مناسب جدًا للـ prototyping

العيوب:
❌ ملفات محلية — لا cloud، لا مشاركة
❌ يحتاج server منفصل في production
❌ البيانات منفصلة عن قاعدة البيانات الرئيسية
❌ لا يدعم Row Level Security (صعوبة في الأذونات)
```

---

### pgvector داخل Supabase (الهدف المستقبلي)

```
pgvector = Extension على PostgreSQL تضيف نوع بيانات vector
Supabase = منصة BaaS (Backend as a Service) مبنية على PostgreSQL

المميزات:
✅ نفس قاعدة البيانات! (artifacts + embeddings في جدول واحد)
✅ Cloud hosted — متاح من أي مكان
✅ يدعم الـ SQL العادي + الـ vector search معًا
✅ Row Level Security مدمج
✅ لا يحتاج server منفصل
✅ يمكن JOIN بين الـ vector results وأي بيانات أخرى مباشرة

العيوب:
❌ أبطأ قليلًا من قواعد متخصصة مثل Pinecone في الـ Scale الضخم
❌ يحتاج إعداد index يدوي (IVFFlat أو HNSW)
```

---

### لماذا pgvector هو القرار الصحيح لمشروعنا؟

لأن حجم البيانات لدينا **محدود** (مئات أو آلاف القطع — لا ملايين).  
في هذا النطاق، فوائد الدمج في نفس قاعدة البيانات تتفوق بكثير على أداء الـ specialized vector DB.

```
المتجهات في Supabase تعني:
سؤال: "أعطني أكثر 3 قطع تشابهًا لهذا السؤال مع اسمها ورابط صورتها"

SQL واحد يجيب على كل شيء:
SELECT name_en, image_url, 1 - (embedding <=> ?) as similarity
FROM artifacts
ORDER BY similarity DESC
LIMIT 3;

بدل: query Chroma → أخذ IDs → query Supabase → merge النتائج
```

```
الزائر يضغط زر الميكروفون
         │
         ▼
```

### لماذا Whisper تحديدًا؟

- مدرَّب على بيانات ضخمة ومتنوعة
- يدعم اللهجات العربية بشكل معقول
- يمكن تشغيله locally أو عبر Groq API (أسرع)
- مجاني أو رخيص جدًا مقارنة بالبدائل

```
         │
         ▼

         │
         ▼
⑥ LLM يُنتج إجابة مبنية على بيانات القطع المسحوبة
         │
         ▼
⑦ الإجابة تظهر للزائر مع روابط للقطع المصادر
```

### تدفق البيانات حسب نوع العملية

#### 📖 زائر عادي يتصفح القطع
```
المتصفح  →  Supabase مباشرة
```
لا يمر على FastAPI — أسرع وأبسط.  
Supabase بيحمي البيانات بنظام **RLS (Row Level Security)** بحيث الزائر يقدر يقرأ بس، مش يعدّل.

#### 🤖 زائر يسأل الـ AI (Chat أو Voice)
```
المتصفح
  → FastAPI
    → Embedding Model (تحويل السؤال لمتجه)
    → pgvector/Chroma (بحث دلالي)
    → Supabase (جيب تفاصيل القطع المتشابهة)
    → LLM (ولّد الإجابة)
  → المتصفح (الإجابة النهائية)
```

#### ✍️ أدمن يضيف قطعة جديدة
```
المتصفح
  → FastAPI (بـ JWT token للتحقق)
    → Supabase Storage (رفع الصورة)
    → Supabase DB (حفظ بيانات القطعة)
    → Embedding Model (توليد المتجه من وصف القطعة)
    → Supabase DB (حفظ الـ embedding في عمود vector)
```

---

### القاعدة العامة — مَن يكلم Supabase مباشرة ومَن يمر على FastAPI؟

```
زائر (قراءة فقط)  →  Supabase مباشرة ✅
أدمن (كتابة/حذف) →  FastAPI أولًا   ✅  (للتحقق من الصلاحيات)
AI / Chat / Voice →  FastAPI حتمًا  ✅  (هو المسؤول عن الـ RAG)
```

> **الفلسفة:** نفصل بين القراءة العامة (سريعة ومباشرة) والعمليات الحساسة (محمية بـ FastAPI والـ JWT).

## 🗄️ تصميم الـ Database — إزاي هنخزن البيانات؟

### العلاقة بين الجداول (ERD مبسط)

```
sections ──────────────────── artifacts
   │                              │
   │  قسم واحد فيه              │  كل قطعة تنتمي لقسم واحد
   │  قطع كتير (1:N)            │
   │                              │
   └──────────────────────────────┘
                                  │
                              embedding  ← عمود إضافي في نفس الجدول
                              (vector)     بيخزن الـ fingerprint الرقمي
                                           للوصف بتاع القطعة
                                           
users (Supabase Auth) ─────── chat_sessions
   │                              │
   │  الأدمن بس عنده             │  كل محادثة مرتبطة
   │  صلاحية الكتابة              │  بـ session_id مؤقت
```

---

### الجدول الأول: `sections` — أقسام المتحف

| اسم العمود | نوع البيانات | وصف |
|-----------|-------------|-----|
| `id` | UUID | معرّف فريد لكل قسم (مولَّد تلقائيًا) |
| `name_en` | TEXT | اسم القسم بالإنجليزية |
| `name_ar` | TEXT | اسم القسم بالعربية |
| `slug` | TEXT (unique) | اسم مبسط للـ URL مثل `ancient-egyptian` |

**البيانات الحالية:**
```
id   │ name_en                      │ name_ar                    │ slug
─────┼──────────────────────────────┼────────────────────────────┼──────────────────
38   │ Ancient Egyptian Antiquities │ الآثار المصرية القديمة     │ ancient-egyptian
39   │ Greco-Roman Antiquities      │ الآثار اليونانية والرومانية │ greco-roman
40   │ Islamic Antiquities          │ الآثار الإسلامية           │ islamic
41   │ Coins Collection             │ مجموعة العملات             │ coins
```

---

### الجدول الثاني: `artifacts` — القطع الأثرية (الأهم)

| اسم العمود | نوع البيانات | وصف |
|-----------|-------------|-----|
| `id` | UUID | معرّف فريد لكل قطعة |
| `name_en` | TEXT | اسم القطعة بالإنجليزية |
| `name_ar` | TEXT | اسم القطعة بالعربية |
| `description_en` | TEXT | وصف تفصيلي بالإنجليزية |
| `description_ar` | TEXT | وصف تفصيلي بالعربية |
| `section_id` | UUID | **Foreign Key** → جدول `sections` |
| `image_url` | TEXT | رابط الصورة في Supabase Storage |
| `period` | TEXT | الحقبة الزمنية (مثلًا: "New Kingdom") |
| `created_at` | TIMESTAMP | تاريخ الإضافة |
| `embedding` | VECTOR(1536) | ⭐ المتجه الرقمي للوصف — يُستخدم في البحث الدلالي |

> **ملاحظة على عمود `embedding`:**  
> هذا العمود هو ما يربط الـ Database بالـ AI.  
> بدل ما نحفظ الـ embeddings في Chroma (ملفات منفصلة)،  
> نحفظها هنا مباشرة بجانب البيانات في نفس السطر.  
> النوع `VECTOR(1536)` يعني متجه بـ 1536 رقم عشري — الحجم الافتراضي لنموذج `text-embedding-3-small`.

---

### الجدول الثالث: `chat_sessions` — سجل المحادثات (اختياري)

| اسم العمود | نوع البيانات | وصف |
|-----------|-------------|-----|
| `id` | UUID | معرّف فريد للرسالة |
| `session_id` | TEXT | معرّف الجلسة (مولَّد في المتصفح، يجمع رسائل الزيارة الواحدة) |
| `role` | TEXT | `'user'` أو `'ai'` |
| `content` | TEXT | نص الرسالة |
| `locale` | TEXT | `'en'` أو `'ar'` |
| `created_at` | TIMESTAMP | وقت الرسالة |

---

### إزاي بيحصل الـ Search بالضبط؟

```
الزائر يسأل: "أثر من عصر الفراعنة مصنوع من الذهب"
                        │
                        ▼
              تحويله لـ embedding:
              [0.12, -0.45, 0.88, ...]  (1536 رقم)
                        │
                        ▼
              مقارنته بكل الـ embeddings في جدول artifacts
              باستخدام Cosine Similarity:
              
              artifact_1 embedding ↔ سؤال  →  similarity = 0.91  ✅ قريب
              artifact_2 embedding ↔ سؤال  →  similarity = 0.73  ✅ قريب
              artifact_3 embedding ↔ سؤال  →  similarity = 0.21  ❌ بعيد
                        │
                        ▼
              رجّع أعلى 5 نتائج تشابهًا
              → جيب وصفهم الكامل من نفس الجدول
              → ابعتهم للـ LLM كـ context
```

---

### الفرق بين طريقتين للبحث

| | Keyword Search (SQL عادي) | Semantic Search (pgvector) |
|--|--------------------------|--------------------------|
| **المثال** | `WHERE name LIKE '%ذهب%'` | `ORDER BY embedding <=> query_vec` |
| **يلاقي** | الكلمة الحرفية بس | المعنى حتى لو الكلمة مختلفة |
| **مثال عملي** | لو قال "gold" مش هيلاقي "ذهب" | هيلاقي "ذهب" حتى لو قال "gold" |
| **الاستخدام** | فلترة عادية | سؤال الـ AI |

---

### ملخص: مين بيكتب في كل جدول؟

```
sections      ← الأدمن فقط (بشكل يدوي نادر)
artifacts     ← الأدمن (إضافة/تعديل/حذف) + AI (يكتب الـ embedding)
chat_sessions ← FastAPI تلقائيًا عند كل رسالة
Supabase Auth ← Supabase نفسها (بيانات تسجيل الدخول)
Storage       ← الأدمن (رفع صور القطع)
```
